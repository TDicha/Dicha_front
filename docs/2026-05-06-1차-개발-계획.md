# Dicha 1차 개발 계획 - 2026-05-06 수정안

## 목표

Dicha Coffee 1차 개발은 현재 구현된 mock 기반 사용자 앱을 실제 API 연동이 가능한 커머스 구조로 정리하는 단계다.

이번 1차의 핵심은 화면을 새로 많이 만드는 것이 아니라, 현재 화면을 유지하면서 데이터 흐름과 구매 흐름을 실제 서비스형 구조로 바꾸는 것이다.

1차에서 우선 완성할 범위는 다음과 같다.

- 상품 목록 조회
- 상품 검색
- 상품 상세 조회
- 상품 옵션 선택
- 장바구니
- 장바구니 선택 상품 구매
- 상품 상세 바로구매
- 회원 구매
- 비회원 구매
- mock 주문 생성
- 회원 주문 조회 구조
- 비회원 주문 조회 구조

1차에서 무리하게 완성하지 않을 범위는 다음과 같다.

- 실제 PortOne 결제 연동
- CSRF 정책 완성
- 아이디 찾기/비밀번호 찾기 전체 인증 플로우
- 배송지 CRUD 전체
- 관리자용 상품/주문 관리 모델

이 항목들은 백엔드 API 계약과 인증/결제 정책이 확정된 뒤 1.5차 또는 2차에서 진행한다.

## 개발 원칙

### 1. page는 mock을 직접 보지 않는다

현재 여러 page가 `@/mock/*`를 직접 import하고 있다. 1차에서는 상품, 장바구니, 주문, 인증 흐름부터 아래 구조로 이동한다.

```txt
page
  -> feature hook
    -> repository
      -> mock adapter or api adapter
        -> mock data or Spring Boot REST API
```

page는 데이터가 mock인지 API인지 알지 않는다. mock/API 전환 책임은 repository와 adapter에 둔다.

### 2. 구매 단위는 product가 아니라 option이다

고객이 실제로 장바구니와 주문에 담는 대상은 `products`가 아니라 `product_options`다.

```txt
products (1) <- (N) product_options
product_options (1) <- (N) carts
product_options (1) <- (N) order_items
```

따라서 `CartItem`과 `OrderItem`은 `productId`만으로 관리하지 않고 `optionId`를 필수로 가진다.

### 3. 회원과 비회원 구매를 모두 지원한다

로그인하지 않아도 구매할 수 있어야 한다.

- 회원: `authStore.user`와 저장 배송지 기반으로 주문 생성
- 비회원: 주문자 이름, 휴대폰 번호, 주문 비밀번호, 배송지 snapshot 기반으로 주문 생성

비회원은 `User`로 만들지 않는다. 비회원 정보는 주문 시점의 snapshot으로 저장한다.

### 4. Zustand와 React Query 책임을 분리한다

Zustand는 사용자가 앱에서 조작 중인 임시 상태를 맡는다.

- 검색 input 값
- 바텀시트 open/close
- 인증 상태 요약
- 장바구니 draft
- checkout draft
- PWA 설치 프롬프트 상태
- 취향 테스트 진행 중 답변

React Query는 서버에서 받아온 원본 데이터를 맡는다.

- 상품 목록
- 상품 상세
- 상품 검색 결과
- 상품 옵션
- 주문 목록
- 주문 상세
- 배송지 목록
- 결제 내역

### 5. Provider는 역할별로 분리한다

현재 `AppProviders`가 QueryClient, PWA 설치 프롬프트 hook, RouterProvider를 함께 가진다.

1차에서는 아래 구조로 분리한다.

```txt
AppProviders
  -> QueryProvider
  -> PwaProvider
  -> RouterProvider
```

- `AppProviders`: provider 조립
- `QueryProvider`: React Query client 생성과 기본 옵션
- `PwaProvider`: `useInstallPrompt` 등 PWA 런타임 이벤트

## 1차 핵심 데이터 모델

### CartItem

현재 구조는 `productId` 기준이라 같은 상품의 다른 옵션을 동시에 담기 어렵다.

변경 후 구조는 다음과 같다.

```ts
type CartItem = {
  cartItemId: string;
  productId: string;
  optionId: string;
  productName: string;
  optionName: string;
  unitPrice: number;
  quantity: number;
  selected: boolean;
};
```

필요 action:

```ts
addItem(item)
updateQuantity(cartItemId, quantity)
removeItem(cartItemId)
toggleSelected(cartItemId)
selectAll()
unselectAll()
clearPurchasedItems(cartItemIds)
clearCart()
```

### CheckoutDraft

장바구니 선택 구매와 바로구매를 명확히 분리하기 위해 `checkoutStore`를 추가한다.

```ts
type CheckoutDraft = {
  mode: "cart" | "direct";
  items: CartItem[];
  ordererType: "member" | "guest" | null;
  guestOrderer?: {
    name: string;
    phone: string;
    orderPassword: string;
  };
  addressSnapshot?: AddressSnapshot;
  paymentMethod?: PaymentMethod;
  pricing: CartPricingSummary;
};
```

구매 진입 흐름:

```txt
상품 상세 바로구매
  -> selected option으로 direct checkout draft 생성
  -> /purchase 이동

장바구니 구매
  -> selected items로 cart checkout draft 생성
  -> /purchase 이동

PurchasePage
  -> checkoutStore.draft 기준으로 렌더링
  -> 회원/비회원 주문자 정보 분기
  -> createOrder()
```

### Order

1차에서는 실제 결제 승인까지 가지 않고, mock adapter라도 반드시 `createOrder()`를 거치게 한다.

```ts
type Order = {
  id: string;
  orderNo: string;
  userId?: string;
  guestPhone?: string;
  status: "order_created" | "payment_completed" | "preparing" | "shipping" | "delivered" | "canceled" | "refunded";
  subtotal: number;
  shippingFee: number;
  discountAmount: number;
  totalAmount: number;
  addressSnapshot: AddressSnapshot;
  paymentMethod: PaymentMethod;
  paymentStatus: "mock_paid" | "pending" | "paid" | "failed" | "canceled" | "refunded";
  orderedAt: string;
};

type OrderItem = {
  id: string;
  orderId: string;
  productId: string;
  optionId: string;
  productName: string;
  optionName: string;
  unitPrice: number;
  quantity: number;
};
```

회원 주문 payload:

```ts
type MemberOrderPayload = {
  ordererType: "member";
  userId: string;
  items: CheckoutItem[];
  addressSnapshot: AddressSnapshot;
  paymentMethod: PaymentMethod;
  pricing: CartPricingSummary;
};
```

비회원 주문 payload:

```ts
type GuestOrderPayload = {
  ordererType: "guest";
  guestOrderer: {
    name: string;
    phone: string;
    orderPassword: string;
  };
  items: CheckoutItem[];
  addressSnapshot: AddressSnapshot;
  paymentMethod: PaymentMethod;
  pricing: CartPricingSummary;
};
```

## 1차에서 제외하거나 후순위로 둘 것

### 실제 PortOne 결제 연동

1차에서는 실제 PG 결제를 붙이지 않는다.

대신 다음 타입과 UI 상태만 준비한다.

```ts
type PaymentMethod = "card" | "kakao_pay" | "naver_pay" | "toss_pay" | "virtual_account";
type PaymentStatus = "mock_paid" | "pending" | "paid" | "failed" | "canceled" | "refunded";
```

주문 생성 mock adapter는 `paymentStatus: "mock_paid"`로 완료 처리한다.

### CSRF 완성

CSRF는 Spring Boot의 세션 방식, 쿠키 정책, Security 설정이 확정된 뒤 구현한다.

1차에서는 다음 정도만 준비한다.

- `endpoints.auth.csrf`
- mutation 요청에 CSRF header를 붙일 수 있는 interceptor 자리
- 토큰은 localStorage에 장기 저장하지 않는다는 원칙

### 계정 찾기

아이디 찾기/비밀번호 찾기는 구매 흐름 완성 뒤 진행한다.

1차 후반에 시간이 남으면 화면 shell과 route만 추가하고, 실제 인증 흐름은 백엔드 정책 확정 뒤 붙인다.

## 1차 완료 기준

아래가 가능하면 1차 개발 완료로 본다.

- 상품 목록을 볼 수 있다.
- 상품을 검색할 수 있다.
- 상품 상세를 볼 수 있다.
- 옵션을 선택할 수 있다.
- 선택한 옵션이 구조화된 장바구니 item으로 들어간다.
- 같은 상품의 다른 옵션을 동시에 장바구니에 담을 수 있다.
- 장바구니에서 선택한 상품만 구매할 수 있다.
- 상품 상세에서 바로구매할 수 있다.
- 로그인하지 않아도 구매할 수 있다.
- 회원 주문과 비회원 주문 payload가 분리된다.
- 주문 완료는 `orderRepository.createOrder()`를 통해 처리된다.
- 완료 화면에 주문번호와 결제 금액이 표시된다.
- 회원은 `/orders`에서 주문 조회 구조를 가진다.
- 비회원은 `/guest-orders`에서 주문 조회 구조를 가진다.
- mock/API 전환 지점이 adapter에 모여 있다.
- build가 통과한다.

## 검증 계획

기본 검증:

```txt
pnpm run build
pnpm run lint
```

현재 lint는 `dev-dist` generated 파일 이슈가 있으므로, 1차 중 `dev-dist` lint 제외 또는 generated 파일 관리 방식을 정리한다.

수동 검증 시나리오:

- 상품 목록 진입
- 상품 검색
- 상품 상세 진입
- 옵션 선택 후 장바구니 담기
- 같은 상품의 다른 옵션 각각 담기
- 장바구니 전체 선택/해제
- 장바구니 일부 선택 구매
- 상품 상세 바로구매
- 비로그인 상태 구매
- 로그인 상태 구매
- 주문 완료 후 주문번호/금액 유지
- 구매 완료 후 구매한 cart item만 제거
- 회원 주문 조회 진입
- 비회원 주문 조회 진입

## 최우선 수정 4개

1. `CartItem` 옵션 구조화
2. `checkoutStore` 추가로 선택 상품 구매 흐름 해결
3. ProductList/Search/ProductDetail의 mock 직접 import 제거
4. PurchasePage 주문 완료를 order repository 기반으로 변경

이 4개를 먼저 끝내면 현재 프로젝트의 가장 큰 구조적 불일치가 정리된다. 이후 비회원 주문 조회, 인증 UX, CSRF/PG 준비를 붙인다.
