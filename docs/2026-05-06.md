## 프로젝트 한 줄 요약

Dicha Coffee는 모바일 퍼스트 PWA 기반의 O2O 커스텀 로스팅 커피 플랫폼 프론트엔드다. 현재는 피그마에서 잡은 주요 사용자 화면을 React/Vite 앱으로 옮긴 상태이며, mock 데이터 기반 화면 구현에서 실제 API 연동이 가능한 feature/repository/hook 구조로 넘어가는 중간 단계다.

## 현재 기술 스택

- Vite
- React 19
- TypeScript
- pnpm
- Tailwind CSS v4
- shadcn/ui 계열 button 컴포넌트

- vite-plugin-pwa
- lucide-react

```txt

  components/
    common/         화면에서 반복되는 UI 조각
    layout/         앱 헤더, 하단 탭, 앱 셸

  features/
    cart/           장바구니 가격 정책, 옵션 표시 정책

    qr/             QR 초기 타입 초안
    notifications/  알림 설정 타입 초안
    기타 도메인     현재는 index 중심의 확장 자리
  pages/            라우트 단위 화면
  services/
    api/            axios client, endpoints, interceptor

```

## 앱 전체 실행 흐름

```txt
src/main.tsx
  -> <App />
    -> <AppProviders />
      -> <QueryClientProvider>
      -> <InstallPromptBootstrap />
      -> <RouterProvider router={router} />
```

- `main.tsx`는 React 앱을 `#root`에 마운트한다.
- `App.tsx`는 화면 로직 없이 `AppProviders`만 반환한다.
- `AppProviders.tsx`는 TanStack Query client를 만들고, PWA 설치 프롬프트 hook과 router를 연결한다.

## Provider 흐름

현재 provider는 `src/app/providers/AppProviders.tsx`에 모여 있다.

```txt
AppProviders
  -> QueryClientProvider
      staleTime: 30초
      retry: 1
  -> InstallPromptBootstrap
      useInstallPrompt()
  -> RouterProvider
      router.tsx
```

무엇을 변경하면 좋은지:

- React Query 기본 캐시 정책을 바꾸려면 `AppProviders.tsx`의 `QueryClient` 옵션을 수정한다.
- 앱 전체 provider가 추가되면 `AppProviders.tsx`에 넣는다.
- PWA 설치 이벤트 처리 방식은 `services/hooks/useInstallPrompt.ts`와 `app/store/installPromptStore.ts`를 같이 본다.

## 라우팅 흐름

현재 라우팅은 `src/app/router/router.tsx`에서 관리한다.

```txt
createBrowserRouter
  -> AppShell
      -> AppHeader
      -> Outlet
      -> BottomTabBar
```

주요 라우트:

- `/`: 홈
- `/products`: 상품 리스트
- `/products/:productId`: 상품 상세
- `/subscription`: 구독
- `/reservation`: 예약
- `/reservation/class`: 클래스 예약
- `/taste-test`: 취향 테스트
- `/cart`: 장바구니
- `/purchase`: 구매
- `/mypage`: 마이페이지
- `/mypage/manage/:sectionId`: 계정 관리 섹션
- `/orders`: 주문 조회
- `/my-blend`: 나의 블렌드
- `/search`: 검색
- `/qr`: QR/O2O
- `/login`, `/signup`

보호 라우트:

```txt
RouteGuard
  -> useAuthStore(status)
  -> authenticated 아니면 /login
  -> authenticated 이면 Outlet
```

현재 보호 대상은 `/mypage`, `/orders`, `/purchase`, `/my-blend` 계열이다.

무엇을 변경하면 좋은지:

- 새 페이지를 추가하려면 `pages/`에 화면을 만들고 `router.tsx`에 route를 추가한다.
- 헤더 제목은 route의 `handle.title`을 바꾼다.
- 헤더/하단 탭을 숨기려면 route의 `handle.chrome = false`를 설정한다.
- 로그인 필요 여부를 바꾸려면 `RouteGuard` children 안팎으로 route 위치를 조정한다.

## Layout 흐름

```txt
AppShell
  -> AppHeader
  -> Outlet(page)
  -> BottomTabBar
```

- `AppHeader`는 `useMatches()`로 현재 route handle을 읽고, `useCartStore()`로 장바구니 수량을 표시한다.
- 홈에서는 DICHA 로고, 검색, 장바구니를 보여준다.
- 일반 페이지에서는 뒤로가기, 페이지 제목, 검색/장바구니를 보여준다.
- `BottomTabBar`는 `bottomTabItems`를 기준으로 홈/쇼핑/예약/마이 탭을 렌더링한다.
- 장바구니와 구매 화면에서는 하단 탭을 숨긴다.

무엇을 변경하면 좋은지:

- 하단 탭 메뉴는 `src/shared/constants/navigation.ts`에서 수정한다.
- route path 상수는 `src/shared/constants/routes.ts`에서 수정한다.
- 헤더 아이콘 노출 조건은 `src/components/layout/AppHeader.tsx`에서 수정한다.
- 공통 앱 껍데기 여백이나 탭 배치는 `AppShell`, `AppHeader`, `BottomTabBar`, `styles`를 같이 본다.

## Store 흐름

현재 전역 상태는 Zustand 기반이며 `src/app/store/`에 있다.

### appStore

역할:

- 검색어
- 바텀시트 open 상태
- QR 입력값과 mock resolve 결과

주요 사용 화면:

- `ProductListPage`
- `SearchPage`
- `ProductDetailPage`
- `QrPage`

변경 포인트:

- 검색어 공유 정책을 바꾸려면 `appStore.searchQuery`와 `SearchInput` 사용 화면을 같이 본다.
- 상품 상세 옵션 바텀시트 상태는 `isBottomSheetOpen`, `openBottomSheet`, `closeBottomSheet`를 사용한다.
- QR 코드가 어디로 이동할지 바꾸려면 `resolveQrCode()`를 수정한다. 장기적으로는 `services/api/endpoints.ts`의 `qr.resolve`와 API service를 연결하는 방향이 맞다.

### authStore

역할:

- 로그인 사용자
- 인증 상태
- 로그인/회원가입/로그아웃 pending/error

흐름:

```txt
LoginPage / SignupPage
  -> useAuthStore.signIn/signUp
    -> services/auth/authService
      -> env.enableMock ? mockUser : apiClient
```

주의점:

- 현재 `env.enableMock` 기본값은 `VITE_ENABLE_MOCK !== "false"`라 환경변수가 없으면 mock 인증이 켜진다.
- 실제 서비스 전환 시 mock 기본값을 dev-only로 제한해야 한다.
- HttpOnly 쿠키 기반 인증을 염두에 두고 `apiClient.withCredentials = true`는 준비되어 있다.
- CSRF 토큰 헤더 처리 흐름은 아직 없다.

### cartStore

역할:

- 장바구니 item 목록
- 수량 변경
- 삭제
- 전체 비우기

현재 CartItem:

```ts
{
  productId: string;
  productName: string;
  quantity: number;
  price: number;
}
```

주의점:

- 현재 옵션명이 `productName` 문자열에 섞여 들어간다.
- 같은 상품의 다른 옵션을 동시에 담기 어렵다.
- 장기적으로 `cartItemId`, `optionId`, `optionName`, `unitPrice`, `selected`를 분리해야 한다.

가격 정책:

- `src/features/cart/cartPricing.ts`
- 쿠폰 할인 5,000원
- 30,000원 미만 배송비 3,000원
- 빈 장바구니는 할인/배송비 0

옵션 표시:

- `src/features/cart/cartItemDisplay.ts`
- 현재는 `productName.split("/")`로 옵션명을 추출한다.
- 이 파일도 장기적으로 mock product detail 의존을 제거해야 한다.

### preferenceStore

역할:

- 취향 테스트 intro/question/result 단계
- 현재 질문 index
- 답변 목록
- 결과 preference

흐름:

```txt
TasteTestPage
  -> tasteQuestions[currentQuestionIndex]
  -> answerQuestion()
  -> completeTest()
  -> tasteProfiles 기반 결과
```

변경 포인트:

- 질문을 바꾸려면 `src/features/taste-test/tasteTestConfig.ts`의 `tasteQuestions`를 수정한다.
- 결과 타입을 바꾸려면 `tasteProfiles`와 `completeTest()`의 매칭 로직을 같이 수정한다.
- 관리자에서 질문을 바꾸게 만들 계획이라면 config가 아니라 API 데이터로 전환해야 한다.

### installPromptStore

역할:

- 브라우저 `beforeinstallprompt` 이벤트 저장
- 설치 배너 dismiss 상태 저장

흐름:

```txt
AppProviders
  -> useInstallPrompt()
    -> beforeinstallprompt 이벤트 저장
    -> installPromptStore.setDeferredPrompt()
```

## Hook 흐름

현재 명시적인 custom hook은 많지 않다.

### useInstallPrompt

파일:

- `src/services/hooks/useInstallPrompt.ts`

역할:

- PWA 설치 이벤트를 가로채서 store에 저장한다.

변경 포인트:

- 설치 배너 UI를 만들 때는 `installPromptStore.deferredPrompt`와 `dismissed`를 읽는 컴포넌트를 추가하면 된다.

### useProducts / useProduct

파일:

- `src/features/products/hooks/useProducts.ts`

흐름:

```txt
page
  -> useProducts(params) / useProduct(productId)
    -> productRepository
      -> env.enableMock ? mockProductAdapter : apiProductAdapter
        -> mockProducts or apiClient
```

현재 상태:

- hook/repository/adapter 골격은 만들어졌다.
- 하지만 `ProductListPage`, `SearchPage`, `ProductDetailPage` 등 주요 page는 아직 `mockProducts`를 직접 import한다.

변경 포인트:

- 상품 API 전환은 page를 직접 고치기보다 `features/products`의 adapter/repository 경유 구조를 완성하는 방식이 좋다.
- 다음 작업은 page의 `@/mock/products` 직접 import를 `useProducts`, `useProduct`로 교체하는 것이다.

### Zustand store hook

`useAppStore`, `useAuthStore`, `useCartStore`, `usePreferenceStore`, `useInstallPromptStore`는 모두 hook처럼 사용된다.

주의점:

- Zustand selector에서 객체를 새로 만들어 반환하면 불필요한 렌더링이나 update depth 문제가 생길 수 있다.
- 4월 작업에서 이미 이 문제를 겪었고, selector를 개별 구독 형태로 분리하는 방향으로 수정했다.

## Props 흐름

현재 props 구조는 비교적 단순하다.

```txt
page
  -> mock/store/local state 계산
  -> common component에 props 전달
```

대표 props:

- `ProductCard({ product })`
- `ProductTileCard({ product, href, compact, showOrigin, showRating, showAddButton, className })`
- `ProfileCard({ user })`
- `Badge({ label })`
- `SearchInput({ value, onChange, placeholder })`
- `BottomSheet({ open, title, onClose, children })`
- `OptionRow({ label, description, price, selected, onClick })`
- `EmptyState({ title, description, action })`
- `SectionTitle({ title, subtitle })`
- `InfoListRow({ label, value })`
- `StatCard({ value, label })`
- `AuthField({ label, icon, ...inputProps })`
- `AuthShell({ badge, title, description, icon, tone, children })`
- `AppCard({ variant, padding, className, ...sectionProps })`
- `PrimaryButton({ ...ButtonProps })`

현재 특징:

- layout 컴포넌트는 props보다 router handle과 store를 직접 읽는다.
- page는 아직 데이터 fetching hook보다 mock import와 local state 계산을 많이 갖고 있다.
- common 컴포넌트는 대부분 표시 전용이며, `value/onChange`, `selected/onClick` 같은 최소 제어 props만 받는다.

변경 방향:

- page에서 계산하는 데이터 출처 로직을 feature hook으로 옮긴다.
- common 컴포넌트는 지금처럼 표시와 작은 interaction props만 받도록 유지한다.
- 도메인 정책은 common 컴포넌트에 넣지 않고 `features/*`로 둔다.

## API / 인증 흐름

현재 API 레이어:

```txt
services/api/client.ts
  -> axios.create({
       baseURL: env.apiBaseUrl,
       withCredentials: true,
       timeout: 10000,
       X-Requested-With
     })
  -> applyInterceptors()
```

endpoints:

- auth: session, login, signup, logout, refresh
- products: list, detail
- orders: list
- subscriptions: list
- qr: resolve

interceptor:

- 401: `인증이 필요합니다.`
- 403: `요청이 허용되지 않았습니다.`
- 기타: `요청 처리 중 문제가 발생했습니다.`

변경 포인트:

- API base URL은 `src/shared/lib/env.ts`의 `VITE_API_BASE_URL` 기준이다.
- 새 API 경로는 `services/api/endpoints.ts`에 먼저 추가한다.
- 실제 요청 함수는 `services/*` 또는 `features/*/adapters/api*.ts`에 둔다.
- 민감한 서버 에러를 그대로 UI에 노출하지 않는 현재 interceptor 방향은 유지한다.

계정 관리 섹션은 `src/shared/constants/accountSections.ts`에 정리되어 있다. 결제수단, 즐겨찾기, 알림, 보안, 앱 설정은 현재 임시 관리 화면 구조다.

### MyBlendPage

```txt
MyBlendPage
  -> mockMyBlendCards
  -> mockMyBlendRecipes
  -> productId 기반 상품 상세 링크
```

나의 블렌드 화면은 구현되어 있으나, 저장/수정/주문 가능 여부 같은 실제 모델은 아직 확정 전이다.

### OrderListPage

```txt
OrderListPage
  -> mockOrders
  -> mockProducts
  -> filter local state
```

주문 상태 필터와 카드형 리스트는 구현되어 있다. 실제 주문 API, 결제 상태, 배송 상태 모델은 아직 연결 전이다.

### QrPage

```txt
QrPage
  -> useAppStore(qrDraft)
  -> resolveQrCode()
  -> product detail 또는 my-blend로 navigate
```

QR 화면에는 카메라 권한, 수동 입력, fallback UI 자리가 있다. 실제 QR 스캔과 서버 resolve는 아직 미구현이다.

## 현재 남아 있는 주요 문제

### 1. README에 merge conflict marker가 남아 있음

`README.md` 상단에 `<<<<<<< HEAD`, `=======`, `>>>>>>>`가 남아 있다. 문서 정리 또는 배포 전 반드시 해결해야 한다.

### 2. mock 인증이 기본값으로 켜짐

`env.enableMock`이 `VITE_ENABLE_MOCK !== "false"`라 환경변수를 설정하지 않으면 mock이 활성화된다. 실제 연동 단계에서는 기본값을 false 또는 dev-only로 바꾸는 것이 안전하다.

### 3. page가 mock을 직접 import하는 곳이 많음

현재 `HomePage`, `ProductListPage`, `ProductDetailPage`, `SearchPage`, `CartPage`, `PurchasePage`, `OrderListPage`, `SubscriptionPage`, `MyBlendPage`, `TasteTestPage` 등에서 mock 직접 참조가 남아 있다.

목표 구조:

```txt
page
  -> feature hook
    -> repository
      -> mock adapter or api adapter
```

### 4. 장바구니 옵션 모델이 약함

현재 CartItem은 `productName` 문자열 안에 옵션명을 섞는다. 이후 실제 주문/결제와 관리자 처리를 생각하면 옵션은 구조화해야 한다.

권장 구조:

```ts
type CartItem = {
  cartItemId: string;
  productId: string;
  optionId: string;
  productName: string;
  optionName: string;
  unitPrice: number;
  quantity: number;
  selected: boolean;
};
```

### 5. 선택 상품 구매 흐름이 불일치함

장바구니 화면은 선택한 상품 기준으로 금액을 계산하지만, 구매 페이지는 전체 cart items를 기준으로 계산한다. `checkoutStore` 또는 `purchaseDraft` 같은 중간 상태가 필요하다.

### 6. 주문 완료가 서버 검증 없이 처리됨

현재 구매 완료는 local state와 `clearCart()`만으로 처리된다. 실제 결제 전까지 mock임을 명확히 하거나, 주문 생성 API 흐름을 추가해야 한다.

### 7. CSRF / 세션 검증 흐름이 아직 없음

쿠키 기반 인증 방향은 좋지만, 실제 Spring Boot 연동 시 다음이 필요하다.

- CSRF 토큰 발급/읽기
- 요청 헤더에 CSRF 토큰 첨부
- CORS allowlist
- SameSite/Secure 쿠키 정책
- 세션 만료 시 UX

### 8. lint 이슈가 기록되어 있음

4월 29일 기록 기준:

- `dev-dist/workbox-5119daf5.js` generated 파일 eslint 문제
- `ProductListPage.tsx`의 `react-hooks/set-state-in-effect` 문제

`dev-dist`는 generated 산출물이라 lint 제외 전략을 먼저 정하는 것이 좋다.

## 오늘 기준 결론

현재 프로젝트는 단순 스켈레톤 단계를 넘어, 졸업작품으로 보여줄 수 있는 사용자 앱 화면 대부분이 구현된 상태다. 다만 아직 핵심 데이터 흐름은 mock과 page 내부 계산에 많이 남아 있어, 실제 서비스형 구조로 가려면 `features/*/hooks -> repository -> adapter` 방향을 각 도메인에 반복 적용해야 한다.
